# calendar-api-spec

Ты помогаешь проекту `Календарь звонков` на стадии API First.

## Основной контекст

- Главный артефакт проекта: `tsp/main.tsp`.
- Артефакт проверки покрытия: `docs/spec-coverage.md`.
- Краткое описание текущего состояния проекта: `README.md`.
- Общие правила работы агентов: `AGENTS.md`.

## Цель навыка

Используй этот навык, когда нужно:

- добавить или уточнить API-сценарий;
- изменить TypeSpec-контракт;
- проверить полноту покрытия сценариев владельца и гостя;
- синхронизировать контракт и сопровождающую документацию;
- провести ревью API-контракта на предмет пропусков и противоречий.

## Обязательные правила

- Всегда отвечай на русском языке.
- Перед выводами перечитывай `AGENTS.md`, `README.md`, `tsp/main.tsp` и `docs/spec-coverage.md`.
- Не придумывай runtime-стек, package manager, команды сборки, тестов или генерации, если они не зафиксированы в репозитории.
- Не выдавай предположения за существующие факты проекта.
- Не изменяй `.github/workflows/hexlet-check.yml`.
- Предпочитай минимальные и точные правки без расширений "на будущее".

## Что считать источником истины

Сначала опирайся на файлы репозитория:

1. `AGENTS.md`
2. `README.md`
3. `tsp/main.tsp`
4. `docs/spec-coverage.md`
5. `.github/workflows/README.md`

Если между файлами есть расхождение, явно укажи его и предложи минимальную синхронизацию.

## Рабочий процесс

1. Определи тип запроса:
   - новый пользовательский сценарий;
   - уточнение существующего поведения;
   - ревью контракта;
   - синхронизация документации.
2. Найди минимальную область изменения:
   - маршруты;
   - модели запросов и ответов;
   - коды ошибок;
   - доменные инварианты.
3. Обнови `tsp/main.tsp` так, чтобы контракт оставался целостным.
4. Проверь, нужно ли обновить `docs/spec-coverage.md`.
5. Проверь, не появилось ли противоречие с `README.md`.
6. В итоговом сообщении кратко перечисли, какие сценарии теперь покрыты или что осталось вне охвата.

## На что обращать особое внимание

- Согласованность имён маршрутов, моделей и alias-типов.
- Полнота кодов ошибок для новых операций.
- Доменные ограничения, например конфликты бронирования и правила занятости.
- Разделение сценариев владельца и гостя.
- Сохранение уровня абстракции API-контракта без ухода в детали реализации.

## Что нужно делать

- проектировать и уточнять endpoint'ы;
- добавлять или корректировать модели запросов и ответов;
- описывать ошибки и контрактные ограничения;
- выявлять пробелы в сценариях и документации покрытия;
- поддерживать согласованность между `tsp/main.tsp`, `docs/spec-coverage.md` и `README.md`.

## Чего делать не нужно

- не придумывать backend-реализацию, frontend-архитектуру, БД или auth без прямого запроса;
- не добавлять локальные команды в документацию, если в репозитории нет подтверждающих конфигов;
- не расширять контракт сущностями, которые не следуют из текущей задачи;
- не менять CI-воркфлоу Hexlet.

## Ожидаемый стиль результата

- Сначала кратко зафиксируй, какой сценарий или пробел обрабатывается.
- Затем внеси минимальные изменения в контракт и документацию.
- После правок коротко объясни, почему изменения достаточны.
- Если данных для точного решения не хватает, задай один короткий уточняющий вопрос вместо догадок.
