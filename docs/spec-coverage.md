# Проверка покрытия TypeSpec-спецификации

## Владелец
- Покрыто: в системе зафиксирован один заранее заданный владелец, который по умолчанию используется в админской части.
- Покрыто: просмотр профиля владельца через `GET /owner/profile`.
- Покрыто: создание типа события через `POST /owner/event-types`, включая передачу `id`, названия, описания и длительности.
- Покрыто: просмотр списка типов событий через `GET /owner/event-types`.
- Покрыто: публикация доступного времени через `POST /owner/availability-windows`.
- Покрыто: просмотр опубликованных интервалов доступности через `GET /owner/availability-windows`.
- Покрыто: просмотр предстоящих встреч через `GET /owner/bookings/upcoming`, где список состоит из будущих бронирований всех типов событий.

## Гость
- Покрыто: просмотр списка видов брони через `GET /event-types`.
- Покрыто: просмотр деталей выбранного типа события через `GET /event-types/{eventTypeId}`.
- Покрыто: просмотр свободных слотов через `GET /event-types/{eventTypeId}/slots`.
- Покрыто: по умолчанию слоты формируются на ближайшие 14 дней от текущей даты.
- Покрыто: если клиент передает `from` и `to`, диапазон должен оставаться внутри того же 14-дневного окна.
- Покрыто: создание бронирования без аккаунта через `POST /bookings`.
- Покрыто: бронирование допускается только на свободный слот из окна ближайших 14 дней.

## Правило занятости
- Покрыто: публичные слоты описаны как вычисляемые из доступности владельца, длительности типа события и уже существующих бронирований.
- Покрыто: при выдаче слотов занятость проверяется глобально для всего календаря, а не только внутри выбранного типа события.
- Покрыто: длительность типа события ограничена положительным значением `durationMinutes >= 1`.
- Покрыто: для публикации доступности и создания бронирования явно зафиксировано правило `startAt < endAt`.
- Покрыто: окна доступности не должны пересекаться между собой; нарушение возвращается как `422 ValidationError`.
- Покрыто: создаваемое бронирование должно совпадать с длительностью выбранного типа события и одному из публично доступных слотов.
- Покрыто: окно записи по умолчанию ограничено ближайшими 14 днями от текущей даты; выход за пределы окна возвращается как `422 ValidationError`.
- Покрыто: конфликт бронирования фиксируется через `409 BookingTimeConflictError` с кодом `booking_time_conflict`.
- Покрыто: правило конфликта сформулировано как глобальное для всего календаря, а не только для одного типа события.
- Покрыто: диапазон запроса слотов при его передаче должен удовлетворять правилу `from < to`; нарушение возвращается как `422 ValidationError`.

## Итог
- Спецификация покрывает сценарии владельца и гостя из текущего ТЗ.
- Спецификация покрывает правило глобальной занятости слота, базовые инварианты временных интервалов и контракт ошибки при конфликте.
- Спецификация остается API-контрактом и не привязана к конкретному стеку реализации.
